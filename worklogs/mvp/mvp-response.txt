       ES-CLI MVP Implementation Plan

       1. TUI Library Decision: Bubble Tea

       Choice: charmbracelet/bubbletea + charmbracelet/bubbles + charmbracelet/lipgloss

       Rationale:
       - Already available in the local Go module cache (v1.3.10)
       - Elm architecture (Model-Update-View) provides clean separation, very testable
       - bubbles/table gives us the index list table out of the box
       - bubbles/textinput for the create-index popup fields
       - bubbles/help for the keybinding help bar in the header
       - lipgloss for K9s-style styling (colored borders, status bars)
       - K9s itself uses tview, but Bubble Tea is more modern, more composable, and far easier to unit test because the entire state is in the Model and the View is a pure function

       2. ES Client Decision: Custom HTTP Client (not official SDK)

       Rationale:
       - The MVP only needs 3 endpoints: _cat/indices?v, PUT /{index}, POST /{index}/_close, DELETE /{index}
       - The official go-elasticsearch client adds complexity and a large dependency for trivial HTTP calls
       - A thin custom client is easier to mock in tests and keeps the binary small
       - Future phases add more endpoints incrementally, and a thin abstraction is easy to extend
       - When Phase 5 (OpenSearch) arrives, the same HTTP client works with a different base URL

       4. View Interface (Extensibility Contract)

       // internal/tui/views/view.go
       package views

       import tea "github.com/charmbracelet/bubbletea"

       type View interface {
           Init() tea.Cmd
           Update(msg tea.Msg) (View, tea.Cmd)
           View() string
           Name() string           // e.g., "index", "node", "shard"
           HelpKeys() []key.Binding // keys to display in header
       }

       Every future phase view (node, shard, dashboard, ILM) implements this interface. The root app.go model holds a currentView View and delegates to it.

       5. Implementation Order (Step by Step)

       Step 1: Project scaffold and Makefile (no TUI yet)

       Files: go.mod, Makefile, docker-compose.yml, cmd/es-cli/main.go (skeleton)

       - go mod init github.com/es-cli/es-cli
       - Dependencies: bubbletea, bubbles, lipgloss
       - docker-compose.yml:
         - elasticsearch:8.19.14 with discovery.type=single-node, xpack.security.enabled=true, ELASTIC_PASSWORD=elastic, ES_JAVA_OPTS=-Xms512m -Xmx512m, container memory limit 2GB, platform linux/amd64 (works on M1
        via Rosetta emulation; alternatively use the official multi-arch image which supports arm64)
         - kibana:8.19.14 linked to ES, ELASTICSEARCH_HOSTS=https://es01:9200
         - Both on same Docker network
       - Makefile targets: make dev-init (docker compose up -d), make dev-start, make dev-stop, make dev-destroy
       - For macOS M1 vs Ubuntu: the official Elastic Docker images are multi-arch since 8.x, so the same compose file works. The only difference is that on macOS, localhost is used to reach the container, while on
        Linux, host.docker.internal may not exist (but localhost also works for Docker port-forwarding). No special handling needed.

       Step 2: Auth module

       Files: internal/auth/auth.go, internal/auth/auth_test.go

       - LoadAuth(path string) (username, password string, err error)
       - Read file, unmarshal JSON as map[string]string
       - Validate: exactly one key-value pair, both non-empty strings
       - Error messages: file not found, invalid JSON, invalid format
       - Tests: valid file, missing file, invalid JSON, empty values, multiple entries (which to pick in MVP - just take the first/only entry)

       Step 3: ES HTTP client

       Files: internal/es/client.go, internal/es/client_test.go

       - Client struct: baseURL, username, password, httpClient *http.Client
       - NewClient(baseURL, username, password string) *Client
       - Skip TLS verification (ES 8.x default uses self-signed certs) or accept a custom CA - for MVP, skip TLS verification with a flag
       - Methods return raw data; parsing is done in domain-specific files
       - Tests use httptest.NewServer to mock ES responses

       Step 4: Index operations

       Files: internal/es/index.go, internal/es/index_test.go

       - type Index struct { Health, Status, Name string; Pri, Rep, DocsCount int; PriStoreSize string }
       - ListIndices() ([]Index, error) - calls GET /_cat/indices?v&format=json&h=health,status,index,pri,rep,docs.count,pri.store.size
         - Using format=json instead of parsing text output - much cleaner
       - CreateIndex(name string, shards, replicas int) error - PUT /{name} with JSON body
       - CloseIndex(name string) error - POST /{name}/_close
       - DeleteIndex(name string) error - DELETE /{name}
       - Tests: mock each endpoint response, test error handling, test JSON parsing

       Step 5: TUI theme and header

       Files: internal/tui/theme.go, internal/tui/header/header.go, internal/tui/header/header_test.go

       - theme.go: lipgloss styles for K9s look (dark background, green/yellow/red for health, blue borders, cyan highlights)
       - header.go: renders 5-line header block
         - Line 1: Logo/title "es-cli" + cluster connection info
         - Line 2: Current view name (e.g., "Indices")
         - Line 3-4: Available keybindings for current view
         - Line 5: Separator line
       - The header accepts helpKeys []key.Binding and viewName string to render dynamically based on active view

       Step 6: Index list view (the main view)

       Files: internal/tui/views/index/model.go, internal/tui/views/index/keybindings.go, internal/tui/views/view.go

       - Uses bubbles/table for the index list
       - Model state: table table.Model, indices []es.Index, sortField string, sortAsc bool, err error, loading bool
       - Init(): fires a tea.Cmd that calls client.ListIndices()
       - Update(): handles key events for sort (Shift+I, Shift+S), open create popup (n), open confirm for close (Shift+C) / delete (Shift+D)
       - View(): renders the table
       - Sorting: by name (alphabetical) or by size (parse size strings like "10kb", "1.2mb" to bytes for comparison)
       - Tests: test sort logic, test Update with key messages, test View output

       Step 7: Confirmation modal

       Files: internal/tui/components/confirm/confirm.go, internal/tui/components/confirm/confirm_test.go

       - On 'y': returns confirmed message; on 'n'/Esc: returns cancelled message

       Key principle: Bubble Tea models are pure functions (Update takes msg, returns model + cmd). This makes them trivially testable without any mocking framework - just construct a model, call Update with a
       message, assert the new state.

       For ES client testing, define a Client interface rather than using the concrete struct directly, so views can accept a mock in tests:

       type ESClient interface {
           ListIndices() ([]Index, error)
           CreateIndex(name string, shards, replicas int) error
           CloseIndex(name string) error
           DeleteIndex(name string) error
       }

       9. Key Architectural Decisions for Extensibility

       1. View interface: Every screen (index, node, shard, dashboard, ILM) implements the same interface. Adding a new view in Phase 1 means creating a new package under internal/tui/views/ and registering it in
       the app router.
       2. Command bar (future-ready): In Phase 1, a :command input bar will be added to app.go. The app will maintain a registry of map[string]ViewFactory where ViewFactory is func(client ESClient) View. For MVP,
       the index view is the only view and loads automatically.
       3. ES client interface: The ESClient interface will grow as new endpoints are needed. Each phase adds methods. The interface is the contract between TUI and backend.
       4. Component reuse: The confirm modal and createindex popup pattern will be reused for node maintenance dialogs (Phase 1), ILM creation (Phase 3), etc.
       5. Auth evolution: In Phase 4 (multi-cluster), the auth module will need to support the richer JSON format. Design the initial LoadAuth to return a struct that can be extended:
       type Credential struct {
           Username string
           Password string
           Host     string  // defaults to "https://localhost:9200" in MVP
       }

       10. Potential Challenges

       - Size parsing for sort: pri.store.size returns human-readable strings like "4.5kb", "1.2gb". Need a parser to convert to bytes for numeric sorting. This should be a utility function with thorough tests.
       - ES 8.x TLS: By default ES 8.x enables HTTPS with self-signed certs. For dev, disable it in docker-compose (xpack.security.http.ssl.enabled=false). For production use, the client should support custom CA or
        skip-verify option.
       - Terminal resize: Bubble Tea sends tea.WindowSizeMsg. All components must handle dynamic resizing. Use lipgloss Width() / Height() to adapt layouts.
       - Table column widths: The bubbles/table needs column widths. Calculate dynamically based on terminal width or use fixed-ratio columns.
