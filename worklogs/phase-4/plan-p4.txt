 Phase 4: Multi-Cluster Support

 Context

 Currently the app connects to a single ES cluster from ES_URL env var. Phase 4 adds multi-cluster support: new auth format with multiple clusters, a cluster selection screen on startup, --cluster flag for direct
 connect, and a second ES dev container.

 Decisions

 - Break old auth format — show migration error if old format detected
 - List selection UI on startup (full-screen, j/k, Enter)
 - Second ES on port 9201, same creds (elastic:elastic)

 New Auth Format (~/.es-cli.auth)

 {
   "local": {
     "username": "elastic",
     "password": "elastic",
     "url": "http://localhost:9200"
   },
   "local-2": {
     "username": "elastic",
     "password": "elastic",
     "url": "http://localhost:9201"
   }
 }

 Keys are cluster names. Each value has username, password, url.

 Changes

 Step 1: Auth — internal/auth/auth.go + test

 New types:
 type ClusterConfig struct {
     Name     string
     Username string
     Password string
     URL      string
 }

 func LoadAuth(path string) ([]ClusterConfig, error)

 - Parse JSON as map[string]map[string]string
 - Iterate keys → cluster name, extract username/password/url
 - Validate each cluster has all 3 fields non-empty
 - If old format detected ({"user":"pass"}), return clear error: "Auth format has changed. Please update ~/.es-cli.auth to multi-cluster format. See docs."
 - Remove old Credential struct

 Old format detection: if any value is a string (not an object), it's the old format.

 Step 2: Cluster Selection TUI — internal/tui/clusterselect/

 New Bubble Tea model (NOT a View — standalone program model for pre-app selection).

 clusterselect.go (~120 lines):
 type Model struct {
     clusters []auth.ClusterConfig
     cursor   int
     width    int
     height   int
 }

 type SelectedMsg struct {
     Cluster auth.ClusterConfig
 }

 type QuitMsg struct{}

 - j/k or up/down moves cursor
 - Enter selects → returns SelectedMsg
 - q/Esc → returns QuitMsg
 - View renders:
   es-cli — Select Cluster

 > local          http://localhost:9200
   local-2        http://localhost:9201

   enter: connect • q: quit

 Step 3: CLI Flag — cmd/es-cli/main.go

 Add --cluster flag using os.Args parsing (no external deps needed):

 func main() {
     clusterName := ""
     for i, arg := range os.Args[1:] {
         if arg == "--cluster" && i+1 < len(os.Args[1:]) {
             clusterName = os.Args[i+2]
         }
     }

     configs, err := auth.LoadAuth(authPath)
     // ...

     if clusterName != "" {
         // Find matching cluster, connect directly
         cluster := findCluster(configs, clusterName)
         runApp(cluster)
     } else if len(configs) == 1 {
         // Single cluster, connect directly
         runApp(configs[0])
     } else {
         // Show cluster selection UI
         selected := runClusterSelect(configs)
         runApp(selected)
     }
 }

 When only 1 cluster in config → skip selection, connect directly.

 Step 4: Header — show cluster name

 Header already has ClusterName field. Currently populated from GetClusterInfo() (ES cluster_name). Change to show the config name (from auth file) instead — more useful since user chose it.

 In NewApp: h.ClusterName = clusterName (passed from main).

 Update NewApp signature:
 func NewApp(client *es.Client, clusterURL, clusterName string) *App

 Step 5: Docker — add es02

 Add to docker-compose.yml:
 es02:
   image: docker.elastic.co/elasticsearch/elasticsearch:8.17.0
   container_name: es-cli-es02
   environment:
     - discovery.type=single-node
     - ELASTIC_PASSWORD=elastic
     - xpack.security.enabled=true
     - xpack.security.http.ssl.enabled=false
     - ES_JAVA_OPTS=-Xms512m -Xmx512m
   ports:
     - "9201:9200"
   deploy:
     resources:
       limits:
         memory: 2G
   healthcheck:
     test: ["CMD-SHELL", "curl -s -u elastic:elastic http://localhost:9200/_cluster/health | grep -q '\"status\"'"]
     interval: 10s
     timeout: 5s
     retries: 30

 Update dev-init in Makefile to wait for both + create sample auth file.
